## 1. SQLite → PostgreSQL 전환

**배경**

**변경 내용**

- BYTEA 데이터형을 통해 바이너리 형태로 대용량 데이터 저장 구조 적용
- DB 접속 정보 `.gitignore`에 추가 (`50e25b4`, 2026-03-24)

**결과/효과**
- VIB 데이터를 바이너리로 직접 저장/조회 가능
- TimescaleDB 확장 경로 확보

## 3. 캐시 전략 폐기 (msgpack → DB 직접 조회)

**배경**
- 기존에는 CSV 파싱 결과를 msgpack으로 캐시하여 조회 성능을 확보
- PostgreSQL 전환 후 모든 CSV 데이터를 DB에 적재하는 구조로 변경되면서 캐시 레이어가 불필요해짐

**변경 내용**

- 고정경로 파일 읽기 기능 제거, 로컬 Path 입력 방식으로 통일 (`e3f83ad`, 2026-03-25)

**결과/효과**
- 캐시 무효화 로직 등 부수 복잡도 제거
- 데이터 흐름 단순화: CSV → DB → API 응답

## 5. 파싱 로직 개선

**배경**
- `ast.literal_eval`로 한 줄씩 평가하는 방식은 Python 인터프리터 의존으로 느림
- DB insert가 건별로 이루어져 대량 데이터 적재 시 병목

**변경 내용**
- `ast.literal_eval` → `json.loads`로 JSON 파싱 전환 (C 확장으로 5~10배 빠름) (`694418f`, 2026-03-23)
- 건별 insert → batch insert 구조로 변경 (`685cc15`, 2026-03-23)
- vib, pulse 파싱 처리 구분으로 로직 명확성 개선 (`027b571`, 2026-03-25)
- TypedDict 정의로 데이터 타입 구체화, dict 구조 개선 (`027b571`, 2026-03-25)

**결과/효과**
- CSV 파싱 성능 5~10배 개선
- batch insert로 DB I/O 횟수 대폭 감소
- TypedDict로 타입 안정성 및 코드 명확성 확보

## 6. 동적 계산 전환 (DB 저장 → 런타임 계산)

**배경**
- tolerance(편차값), high_vib_event 임계값 등이 유동적으로 변경될 수 있는 값
- DB에 고정 저장하면 임계값 변경 시 전체 재계산이 필요

**변경 내용**
- tolerance 기반 `is_valid` 체크 삭제, 동적 계산으로 전환 (`0555c0a`, 2026-03-24)
- `high_vib_event`도 DB 저장 제외, 런타임 계산값으로 변경 (`b2ac66f`, 2026-03-24)

**결과/효과**
- 임계값 변경 시 재적재 없이 즉시 반영
- DB 스키마 단순화, 저장 데이터량 감소
